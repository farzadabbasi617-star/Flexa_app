- TypeScript بدون خطاست.
- 139 تست واحد در 19 فایل همگی موفق‌اند.

- دو مورد با اولویت بالا شناسایی شد: خروج اطلاعات خصوصی بازیکنان از API عمومی و نبود E2E واقعی برای جریان‌های مالی/دوکاربره.

## یافته‌ها

### P0 / High — اطلاعات خصوصی در API عمومی بازیکنان

`GET /api/players` بدون ورود این فیلدها را برمی‌گرداند:

- `email`
- `visibleUserId` (UUID داخلی کاربر)
- game IDs و usernameهای بازی‌ها
- role و شناسه Gament

صفحه عمومی لیست بازیکنان به email و UUID داخلی نیازی ندارد. پیشنهاد:

- پاسخ Public فقط شامل id بازیکن، displayName، username، avatar، rating، wins/losses، level و verification عمومی باشد.
- اطلاعات مالکیت فقط برای خود کاربر یا admin برگردد.
- email هیچ‌وقت در API عمومی بازیکنان قرار نگیرد.

### P0 / Coverage Risk — نبود E2E مالی و دوکاربره

بدون دو حساب QA واقعی و نشست admin نمی‌توان این مسیرها را به شکل End-to-End در Production تست کرد:

- ورود و مصرف واقعی OTP بازیابی
- شارژ/برداشت کیف پول و بررسی پنل ادمین
- ثبت‌نام پولی و Double-click هم‌زمان
- اتصال دو حساب تلگرام، صف 1V1 و تبادل دو QR
- نتیجه، اعتراض و پرداخت جایزه 80,000 تومان
- Escrow فروشگاه و Refund

منطق تراکنش و تست‌های واحد بررسی شد، ولی برای تأیید نهایی باید دو حساب QA اختصاصی ساخته شود.

### P1 — Pagination نامعتبر و امکان Query بدون سقف

نتایج تست:

- `/api/players?page=-5&limit=100000` → 500
- `/api/tournaments?page=-2&limit=100000` → 500
- پارامتر `abc` در players/tournaments ممکن است Limit را عملاً حذف کند.
- Store برای پارامترهای `abc` مقدار `page:null` و `pageSize:null` برمی‌گرداند.

پیشنهاد: utility مشترک `safePagination()` با page حداقل 1، limit پیش‌فرض و سقف 100.

### P1 — هدرهای امنیتی ناقص

هدرهای موجود:

- `X-Frame-Options: SAMEORIGIN`
- `X-Content-Type-Options: nosniff`
- Referrer Policy و Permissions Policy

موارد غایب:

- `Strict-Transport-Security`
- `Content-Security-Policy` یا حداقل `Content-Security-Policy-Report-Only`

### P1 — تنظیمات Integration ناقص

نسخه زنده:

- `/api/telegram/setup` → 503 چون `TELEGRAM_SETUP_SECRET` تنظیم نشده است.
- integration پیش‌ثبت‌نام قدیمی → 503 چون `TELEGRAM_INTEGRATION_SECRET` تنظیم نشده است.

Webhook اصلی سالم است؛ این‌ها قابلیت‌های مدیریتی/Legacy هستند، ولی وضعیتشان باید در مستندات مشخص یا Envها تنظیم شود.

### P1 — endpoint دیباگ عمومی

`GET /api/debug/images` بدون احراز هویت همه رکوردهای `site_images` را برمی‌گرداند. داده فعلی حساس نیست، اما endpoint دیباگ نباید در Production عمومی باشد؛ حذف یا admin-only شود.

### P1 — اخطارهای Lint و رندرهای اضافی

ESLint:

- 0 Error
- 95 Warning در 52 فایل
- 48 مورد `<img>` به جای `next/image`
- 46 مورد setState هم‌زمان داخل effect
- 1 مورد `Date.now()` در Render

این موارد بیشتر Performance/Maintainability هستند، ولی 46 effect می‌توانند رندر اضافی ایجاد کنند.

### P1 — تصاویر بسیار بزرگ

نمونه‌ها:

- `icon-teams.png`: حدود 3.1MB
- `icon-judging.png`: حدود 2.7MB
- چند فایل لوگو: 0.9 تا 1.3MB
- چند آواتار: حدود 0.4MB

پیشنهاد: تبدیل به WebP/AVIF، رزولوشن متناسب و حذف source artwork از public در صورت عدم نیاز Runtime.

### P2 — فایل‌های بزرگ و پیچیدگی

- Telegram webhook هنوز حدود 159KB است.
- صفحه جزئیات تورنمنت حدود 49KB است.
- Schema حدود 53KB است.

ماژولارکردن شروع شده، اما wallet/registration/callback/admin Telegram باید بیشتر جدا شوند.

### P2 — هشدار SSL آینده PostgreSQL

Render هشدار می‌دهد که `sslmode=require` در نسخه آینده pg semantics ضعیف‌تری خواهد داشت. مقدار پیشنهادی:

```text
sslmode=verify-full
```

### P2 — UX صفحه بازیکنان

فرم «افزودن بازیکن» برای همه کاربران نمایش داده می‌شود، اما API فقط admin را قبول می‌کند. فرم باید فقط برای admin نمایش داده شود یا کلاً به پنل admin منتقل شود.

### P2 — فروشگاه بدون داده

`/api/store/listings` سالم است ولی فعلاً `items: []` دارد. این خطای فنی نیست؛ برای تست کامل خرید/فروش باید داده QA اضافه شود.

## برنامه اصلاح پیشنهادی

1. حذف email و UUID مالک از API عمومی players و اصلاح Pagination.
2. ساخت دو حساب QA و اجرای سناریوی کامل 1V1/Wallet/Prize/Refund.
3. محافظت از debug endpoint و افزودن HSTS/CSP Report-Only.
4. افزودن تست Integration برای auth، wallet، registration، store escrow و Telegram queue.
5. رفع تدریجی 95 Warning و بهینه‌سازی تصاویر بزرگ.
6. ادامه تفکیک webhook و سرویس‌های مالی از Routeهای بزرگ.

## محدودیت این دور تست

هیچ عملیات مخرب یا مالی واقعی روی Production اجرا نشد. برای فاز دوم نیاز است:

- دو حساب QA معمولی و متصل به دو Telegram ID متفاوت
- یک حساب QA admin
- موجودی آزمایشی قابل Refund
- اجازه ایجاد Listing/Tournament آزمایشی

این حساب‌ها باید اختصاصی QA باشند؛ رمز حساب واقعی یا Secretها نباید در چت ارسال شوند.
